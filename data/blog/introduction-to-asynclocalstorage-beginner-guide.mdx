### Introduction

In large Node.js applications, managing request-specific context across asynchronous calls can be tricky. That’s where AsyncLocalStorage (introduced in Node.js 13) comes in. It provides a way to store and access data throughout the lifetime of an asynchronous operation — like request IDs, user sessions, or transaction contexts.

In this guide, we’ll explore how AsyncLocalStorage works with real project-like examples.

### Why Use AsyncLocalStorage?

- Request Tracking: Attach a unique request ID and log it everywhere.
- Context Sharing: Pass user/session data without threading it manually.
- Clean Code: Avoid adding parameters everywhere just to propagate context.

#### Step 1: Setup AsyncLocalStorage Context

```js
// context.js
import { AsyncLocalStorage } from 'node:async_hooks'

const asyncLocalStorage = new AsyncLocalStorage()

export function runWithContext(store, callback) {
  return asyncLocalStorage.run(store, callback)
}

export function getContext() {
  return asyncLocalStorage.getStore()
}
```

Here we:

- Create a global AsyncLocalStorage instance.
- Wrap requests in runWithContext.
- Provide a helper getContext to fetch current context.

#### Step 2: Add Request Context in Express

```js
// index.js
import express from 'express'
import { runWithContext, getContext } from './context.js'
import { logInfo } from './logger.js'
import { randomUUID } from 'node:crypto'

const app = express()

app.use((req, res, next) => {
  const store = { requestId: randomUUID(), url: req.url }
  runWithContext(store, () => next())
})

app.get('/', (req, res) => {
  logInfo('Hello World!')
  res.send('Check logs for request context!')
})

app.listen(3000, () => console.log('Server running on http://localhost:3000'))
```

What happens:

- Each request gets a unique requestId.
- We store it inside the context.
- All logs now automatically include it.

#### Step 3: Logging with Context

```js
// logger.js
import { getContext } from './context.js'

export function logInfo(message) {
  const ctx = getContext()
  const prefix = ctx ? `[${ctx.requestId}]` : '[no-context]'
  console.log(`${prefix} ${message}`)
}
```

Now every log has the request ID:

```bash
[9a7f1a2d-1234] Hello World!
```

#### Step 5: Direct AsyncLocalStorage Usage Example

Sometimes, you may want to access the context directly inside a deeply nested async function. Here’s how you can do it:

```js
// index.js
[...]
async function deepAsyncOperation() {
  await new Promise((resolve) => setTimeout(resolve, 100));
  const ctx = getContext();
  logInfo(`Deep async operation for request: ${ctx?.requestId}`);
}

app.get("/deep", async (req, res) => {
  await deepAsyncOperation();
  res.send("Deep async operation done");
});
[...]
```

Example response in your terminal logs:

```bash
[c2a1e7b0-5678] Deep async operation for request: c2a1e7b0-5678
```

This demonstrates that even in deeply nested async calls, you can reliably fetch the context using `getContext()`.

### AsyncLocalStorage and OpenTelemetry

If you've used OpenTelemetry's Node.js SDK, you've already been relying on AsyncLocalStorage without necessarily realizing it. The standard context manager for Node — `AsyncLocalStorageContextManager` from `@opentelemetry/context-async-hooks` — is built directly on top of the same primitive this guide walks through. That's how OpenTelemetry knows which span is "active" as your code moves through async operations: the same mechanism that propagates your `requestId`.

You can combine the two directly — enrich the active OpenTelemetry span with your own request context:

```js
// logger.js (or anywhere inside your request-scoped code)
import { trace } from '@opentelemetry/api'
import { getContext } from './context.js'

export function annotateActiveSpan() {
  const ctx = getContext()
  const span = trace.getActiveSpan()
  if (ctx && span) {
    span.setAttribute('request.id', ctx.requestId)
  }
}
```

Since both your custom context and OpenTelemetry's active-span tracking ride on the same `AsyncLocalStorage` instance internally, calling this anywhere inside the request lifecycle reliably tags the current trace — no extra propagation logic needed.

### Testing Code That Uses AsyncLocalStorage

Since `getContext()` returns `undefined` outside of a `run()` call, any test that exercises code depending on it needs to set up that context itself — the same way a real request would:

```js
import { runWithContext } from './context.js'
import { logInfo } from './logger.js'

test('logInfo includes the request ID from context', () => {
  const store = { requestId: 'test-request-id' }

  runWithContext(store, () => {
    // Anything called synchronously or asynchronously inside this
    // callback sees the same context a real request would.
    logInfo('test message')
  })
})
```

If you forget this and call `logInfo()` directly in a test with no wrapping `runWithContext`, you'll get the `[no-context]` fallback rather than a test failure — worth adding an explicit assertion on the logged output if you want a missing context to actually fail the test rather than silently degrade.

### Common Pitfalls

A few things that catch people out once they start relying on this in production:

- **Context doesn't cross `worker_threads` boundaries.** Each worker thread runs in its own isolate with its own event loop — `AsyncLocalStorage` state doesn't survive `postMessage`. If a worker needs the request context, pass the specific fields you need explicitly when you spawn or message it.
- **Everything must run inside the `run()` callback.** If you kick off a "fire and forget" operation _outside_ the callback passed to `asyncLocalStorage.run()` — for example, scheduling work from a module-level event listener set up at startup — that code never had context to inherit in the first place. `getContext()` will return `undefined`, not stale data, so this fails loudly rather than silently, but it's worth knowing why.
- **It's not free, but it's cheap.** Modern Node's `AsyncLocalStorage` (backed by `async_hooks`) has measurably low overhead for typical request-handling workloads. It's not zero — avoid creating a new store per-item in a tight loop processing thousands of records — but for one store per HTTP request, the cost is negligible.

### Conclusion

AsyncLocalStorage is a powerful but underused feature in Node.js. It makes context handling elegant and less error-prone. If you’ve ever struggled passing IDs or user data across layers, this tool can save you.

Next time you build middleware-heavy apps, try AsyncLocalStorage for clean and contextual logging.
